Overview
Welcome to the Lab
In the session you watched a bastion host get hardened, a jump server used to pivot, and a redundant bastion added behind a tradeoff discussion. In this lab, you build all of it yourself, end to end, on your own machine. You'll stand up a small multi-zone network, harden a bastion from scratch, prove a brute-force attack fails after hardening, review real logs, and add a redundant bastion behind a load balancer.

Estimated time: 2.5–3.5 hours, in one sitting or spread across a few. Format: fully independent — work top to bottom, check off each checkpoint as you go, and use the sidebar to track your progress. Nothing here touches a cloud account or production system — it's all local Docker containers.

Prerequisites

Before starting Part 0, confirm you have:

• Docker and Docker Compose installed and working (docker --version and docker compose version both return something)

• A terminal (macOS/Linux Terminal, or Windows WSL2 / Git Bash)

• Basic comfort running shell commands

Part 0
Build the Topology
~20 min
You're going to create four containers on three isolated Docker networks, mimicking the trust zones from the slides: 
[ attacker/redteam ] --> [ bastion (dmz) ] --> [ jump (internal) ] --> [ appserver (data) ]

Step 1 — Create a project folder

mkdir -p ~/secure-access-lab && cd ~/secure-access-lab
mkdir keys

Step 2 — Create docker-compose.yml

Create this file with exactly this content:

services:
  bastion:
    image: rastasheep/ubuntu-sshd:18.04
    container_name: bastion
    networks:
      dmz:
        ipv4_address: 172.28.0.10
      internal:
        ipv4_address: 172.28.1.10
    cap_add:
      - NET_ADMIN
    ports:
      - "2222:22"

  jump:
    image: rastasheep/ubuntu-sshd:18.04
    container_name: jump
    networks:
      internal:
        ipv4_address: 172.28.1.20
      data:
        ipv4_address: 172.28.2.20

  appserver:
    image: rastasheep/ubuntu-sshd:18.04
    container_name: appserver
    networks:
      data:
        ipv4_address: 172.28.2.30

  redteam:
    image: rastasheep/ubuntu-sshd:18.04
    container_name: redteam
    networks:
      dmz:
        ipv4_address: 172.28.0.50

networks:
  dmz:
    ipam:
      config:
        - subnet: 172.28.0.0/24
  internal:
    ipam:
      config:
        - subnet: 172.28.1.0/24
  data:
    ipam:
      config:
        - subnet: 172.28.2.0/24

Notice: redteam and appserver are never on the same network. Only bastion bridges every zone except data, and only jump bridges into data. That's the architecture doing its job before you've configured anything.

Step 3 — Bring it up

docker compose up -d
docker compose ps

All four containers should show Up. If any shows Restarting, run docker compose logs <name> and fix before continuing.

Step 4 — Prove the segmentation works

From redteam, try to reach appserver directly:

docker exec -it redteam ping -c 2 172.28.2.30

This should fail — no route exists between the dmz and data subnets. Write down what error you see.

Checkpoint 0
Part 1
Harden the Bastion
~40 min
Right now bastion has Ubuntu's defaults: password auth enabled, root login allowed, no rate limiting. You're going to fix all of it.

Step 1 — Generate a real keypair on your own machine

ssh-keygen -t ed25519 -f ~/secure-access-lab/keys/bastion_key -C "intern-bastion-key" -N ""

Step 2 — Create a non-root user and install your key

docker exec -it bastion bash -c "useradd -m -s /bin/bash intern && mkdir -p /home/intern/.ssh"
docker cp ~/secure-access-lab/keys/bastion_key.pub bastion:/home/intern/.ssh/authorized_keys
docker exec -it bastion bash -c "chown -R intern:intern /home/intern/.ssh && chmod 700 /home/intern/.ssh && chmod 600 /home/intern/.ssh/authorized_keys"

Step 3 — Edit the SSH daemon config

docker exec -it bastion bash
apt-get update && apt-get install -y nano fail2ban
nano /etc/ssh/sshd_config

Set (or add) these lines exactly:

PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AllowUsers intern
LogLevel VERBOSE

Save, then restart SSH:

service ssh restart

Step 4 — Configure fail2ban

Still inside the bastion shell:

cat > /etc/fail2ban/jail.local << 'EOF'
[sshd]
enabled = true
port = ssh
maxretry = 3
findtime = 120
bantime = 600
EOF
service fail2ban restart
exit

Step 5 — Test key login from your own machine

ssh -i ~/secure-access-lab/keys/bastion_key -p 2222 intern@localhost

You should land on the bastion. Type exit to leave.

Checkpoint 1
Part 2
Attack Simulation, Before vs. After
~30 min
This is the part that proves hardening actually matters, not just that you followed steps.

Step 1 — Install a brute-force tool on redteam

docker exec -it redteam bash -c "apt-get update && apt-get install -y hydra"

Step 2 — Attempt a password-guessing attack

docker exec -it redteam bash -c "echo -e 'root\nadmin\npassword\n123456' > /tmp/users.txt && echo -e 'password\n123456\nadmin\nletmein' > /tmp/pass.txt"
docker exec -it redteam hydra -L /tmp/users.txt -P /tmp/pass.txt ssh://172.28.0.10

Step 3 — Record the result

Because you disabled password authentication and root login in Part 1, this attack should fail entirely. Save the terminal output showing the failed attempt — you'll need it for submission.

Step 4 — Check fail2ban's view of this

docker exec -it bastion fail2ban-client status sshd

You should see a non-zero failed-attempt count, even though nothing succeeded — fail2ban watches every attempt, not just successful ones.

Checkpoint 2
Part 3
Pivot Through the Jump Server
~30 min
Now you'll connect through the bastion to reach the app server — the actual point of a jump-server architecture.

Step 1 — Harden the jump server

Repeat the key-generation and hardening steps from Part 1 on the jump container (same AllowUsers / PasswordAuthentication no / key-install pattern). Use a separate keypair: ~/secure-access-lab/keys/jump_key.

Step 2 — Confirm jump is unreachable directly

jump has no published port and isn't on a network your laptop can reach. The only way in is through bastion.

Step 3 — Use ProxyJump

ssh -i ~/secure-access-lab/keys/jump_key -o ProxyJump="intern@localhost:2222 -i ~/secure-access-lab/keys/bastion_key" intern@172.28.1.20

If that connects, you're now on jump having authenticated twice — once at the bastion, once at the jump server — without ever installing a VPN client.

Step 4 — Reach appserver from jump

ping -c 2 172.28.2.30

Part 4
Read the Logs Like an Analyst
~25 min

Step 1 — Pull the authentication log

docker exec -it bastion bash -c "grep sshd /var/log/auth.log | tail -40"

Step 2 — Identify four fields in at least three lines

For your hydra attempt and your own successful key login, write down: (1) timestamp, (2) source IP, (3) username attempted, (4) outcome (accepted / failed / invalid user).

Step 3 — Write your analysis

In 3–4 sentences: if you were a security analyst seeing only this log, how would you tell your own legitimate login apart from the hydra attack? What pattern gives it away?

Checkpoint 4
Part 5
Add Redundancy, Then Break It
~35 min
This is the redundancy-vs-security exercise, but hands-on instead of on paper.

Step 1 — Add a second bastion

Add to your docker-compose.yml: IP 172.28.0.11 on dmz and 172.28.1.11 on internal, container name bastion2, published port 2223:22.

docker compose up -d bastion2

Step 2 — Harden bastion2

Use the exact same steps as Part 1, but with a different keypair (bastion2_key) — never share one key across redundant hosts.

Step 3 — Add HAProxy

Create haproxy.cfg:

frontend ssh_front
    bind *:2200
    mode tcp
    default_backend ssh_back

backend ssh_back
    mode tcp
    balance roundrobin
    server bastion1 172.28.0.10:22 check
    server bastion2 172.28.0.11:22 check

Add an haproxy service using the haproxytech/haproxy-alpine image, mounting this config, exposing port 2200.

Step 4 — Test failover

docker compose stop bastion
ssh -i ~/secure-access-lab/keys/bastion2_key -p 2200 intern@localhost

You should still get in — HAProxy routed you to bastion2.

Step 5 — Reflect

What did you just gain (availability), and what did you just take on (a second host that must be patched, monitored, and have its own key managed forever)? Reference the four-factor framework (criticality, compliance, attacker-foothold cost, recovery time) for your hypothetical scenario.

Checkpoint 6

Troubleshooting

"Cannot connect to the Docker daemon" — Docker isn't running. Start it and retry.
Connection refused on port 2222 — the bastion container isn't up yet, or compose restarted. Check docker compose ps.
Key login still asks for a password — confirm PasswordAuthentication no was saved and you ran service ssh restart.
ProxyJump fails with a permission error — check authorized_keys ownership and permissions (700 on .ssh, 600 on authorized_keys).
HAProxy won't start — check config syntax with haproxy -c -f /usr/local/etc/haproxy/haproxy.cfg inside the container.
